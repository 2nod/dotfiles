# Meeting Agenda

## 基本方針

アジェンダは、情報の要約ではなく会議を進めるための道具として作る。
本文は短くし、参加者が「何を見て、何を話し、何を決めるか」を追える形にする。

既存ページを直す場合は、末尾に追記しない。
目的、前提、アジェンダ、確認事項、出口の関係が通るように本文全体を組み替える。

日本語で書くときは `writing/japanese-tech-writing` も使う。
一文ごとに改行し、冗長な言い換えと空虚な強調を削る。

## 作業手順

1. 会議の目的、日時、参加者、決めたいことを特定する。
2. Slack、Notion、PR、mock、dev 環境など、ユーザーが示した材料を読む。
3. 材料を「体験を見るもの」「実装を確認するもの」「判断材料ではなく参考に留めるもの」に分ける。
4. 確定事実、解釈、未確認事項を混ぜない。
5. `templates/simple-agenda.md` を土台に、不要な節を削り、必要な節だけ残す。
6. 既存の Notion ページを更新する場合は、更新前に fetch して現在の本文を確認する。
7. 更新後に再 fetch して、目的、前提、確認事項、出口に矛盾がないか確認する。

## 判断の分け方

mock や prototype は、体験仮説と導線確認の材料として扱う。
実装境界、責務分担、API contract、package 境界は、PR、dev 環境、実装済みコードを材料に確認する。

Slack thread の途中返信は、独立した追記にしない。
既存の目的や確認事項に吸収し、必要なら「事前に見るもの」や「確認すること」にリンクを置く。

## 書き方

目的は、会議で扱う問いを一つか二つに絞って書く。
前提は、議論に必要な事実だけを書く。
アジェンダは、時間、内容、ゴールの三列を基本にする。
確認事項は、参加者がその場で判断できる粒度にする。
出口は、会議後に残る成果物と次の行動を書く。

抽象的な見出しを避ける。
「話すこと」だけで足りない場合は、「体験の確認」「実装の確認」のように材料の種類が分かる見出しにする。
